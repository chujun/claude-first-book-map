  OpenSec 优缺点

  优点

  - 安全内嵌：威胁建模前置，避免后期修复成本
  - 合规性好：符合 ISO 27001、OWASP 标准
  - 风险可视化：早期识别安全风险

  缺点

  - 开发周期长：安全验证增加额外步骤
  - 团队要求高：需要安全专业知识
  - 灵活性受限：流程较重，不适合快速原型

  建议

  如果你正在开发 book-map
  这类展示型应用（数据可视化、无敏感交易），安全流程可以适度简化。建议采用
  轻量级安全检查 + 快速迭代 的混合模式。

  简单来说

  对于 book-map 这类应用，安全验证主要指：

  1. 用户输入 - 搜索框、筛选器是否过滤特殊字符
  2. 外部数据 - 爬取的书籍数据是否做安全处理
  3. 部署配置 - 服务器是否暴露敏感端口

  实际检查清单

  - [ ] 无硬编码密码/密钥
  - [ ] 用户输入已验证
  - [ ] 错误信息不泄露系统细节
  - [ ] 第三方依赖无高危漏洞

  对于你的项目，因为是纯展示型（只读数据 +
  前端渲染），安全风险较低，主要关注依赖库漏洞扫描即可。

  四、项目选型建议

  book-map 场景分析

  项目类型: 数据展示型
  复杂度: 中低
  团队规模: 小（你一人或小团队）
  更新频率: 低~中
  安全要求: 低（公开数据）

  推荐方案

  ┌────────┬────────────────┬──────────────────────────────────────────────┐
  │ 优先级 │      方案      │                     理由                     │
  ├────────┼────────────────┼──────────────────────────────────────────────┤
  │ 主推   │ CLA + 轻量安全 │ AI 工具链加持 + 基础安全检查，平衡效率与质量 │
  ├────────┼────────────────┼──────────────────────────────────────────────┤
  │ 备选   │ Agile + TDD    │ 需求变化多时切换                             │
  ├────────┼────────────────┼──────────────────────────────────────────────┤
  │ 不推荐 │ OpenSec        │ 过于重量，安全要求不对口                     │
  └────────┴────────────────┴──────────────────────────────────────────────┘

  CLA + 轻量安全具体流程

  1. 需求收集 → features.json 管理
  2. Plan 模式规划 → 拆解任务
  3. TDD 实现核心功能
  4. Code Review 审核
  5. 轻量安全检查:
     - 依赖漏洞扫描 (npm audit / safety)
     - 无硬编码密钥
     - 输入验证
  6. Git 提交 → 自动部署（可选）

  ---
  五、最终建议

  对于 book-map 项目：

  1. 采用 CLA 工作流 作为主体流程
  2. 叠加轻量安全检查（依赖扫描 + 基础验证）
  3. TDD 用于核心功能（聚类、搜索等）
  4. Agile 思维管理需求（短迭代，快速交付）

  这样既能享受 AI 工具链的高效，又不失质量保障，且不过度复杂。

  ---
  OpenSpec vs 其他 AI 开发流程选型对比

  一、OpenSpec 核心定位

  GitHub: https://github.com/Fission-AI/OpenSpec

  OpenSpec 是一个轻量级规范框架，核心理念：

  fluid not rigid         — 流畅不僵硬
  iterative not waterfall — 迭代不瀑布
  easy not complex       — 简单不复杂
  brownfield-first       — 支持存量项目，不只是新建项目

● OpenSpec 与 TDD 有机结合

  一、两者天然契合点

  1. Scenario 即测试用例

  OpenSpec 的 specs 使用 Given/When/Then 格式：

  ### Requirement: 用户登录
  系统必须在有效凭证提交后发放 JWT 令牌。

  #### Scenario: 有效凭证登录
  - GIVEN 已注册用户
  - WHEN 提交正确用户名和密码
  - THEN 返回 JWT 令牌并重定向到仪表盘

  #### Scenario: 无效凭证登录
  - GIVEN 未注册用户
  - WHEN 提交错误密码
  - THEN 显示错误信息且不发放令牌

  这些 scenario 可直接转化为测试用例！

  五、实际结合示例

  场景：添加暗色主题

  1. /opsx:propose add-dark-mode

  2. 编写 specs/dark-mode/spec.md：

     ### Requirement: 暗色主题切换
     用户应能在亮色和暗色主题间切换

     #### Scenario: 切换到暗色
     - GIVEN 用户在亮色主题
     - WHEN 点击主题切换按钮
     - THEN 界面变为暗色主题

     #### Scenario: 记住主题偏好
     - GIVEN 用户选择暗色主题
     - WHEN 刷新页面
     - THEN 仍显示暗色主题

  3. TDD 实施：
     - 写测试: test_toggle_theme.py
     - 最小实现: 仅 CSS 变量切换
     - 重构: 提取 ThemeContext

  4. /opsx:verify (检查测试通过)
  5. /opsx:archive

  六、Schema 定制：TDD-OpenSpec 融合

  可以创建自定义 schema：

  # openspec/schemas/tdd-driven/schema.yaml
  name: tdd-driven
  artifacts:
    - id: proposal
      generates: proposal.md
      requires: []

    - id: specs
      generates: specs/**/*.md
      requires: [proposal]
      # Scenario 测试化规则

    - id: design
      generates: design.md
      requires: [proposal]

    - id: tasks
      generates: tasks.md
      requires: [specs, design]
      # 包含测试编写子任务

    - id: tests
      generates: tests/**/*.spec.ts
      requires: [specs]
      # TDD: 测试先于实现

  八、结论

  OpenSpec 和 TDD 可以完美结合：

  - OpenSpec 提供了 需求规范化和变更管理
  - TDD 提供了 实现质量保证
  - 两者互补：OpenSpec 的 Scenario 就是 TDD 的测试用例

  推荐工作流：
  /opsx:propose → 编写含 Scenario 的 specs → TDD 红→绿→重构 → /opsx:verify →
  /opsx:archive
